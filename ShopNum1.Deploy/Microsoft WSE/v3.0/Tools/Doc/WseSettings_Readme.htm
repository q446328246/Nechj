Welcome to the Web Services Enhancements 2.0 Settings

The Web Services Enhancements 2.0 Settings is a tool to assist developers to configure applications which use the Web Services Enhancements (WSE).  For more information on WSE, visit us on the Web.

How to invoke the tool

Step 1: Start Visual studio;

Step 2: Open the project which you want to use Web Services Enhancement.

Step 3: Right-click on the project name, on the popup menu, click on the "WSE Settings 2.0..." command.

 

                           Figure 1: right click on the WseSettings

Step 4: Switch between the tabs to adjust settings for WSE.

Step 5: Click "OK" to save all the settings.

 

What this tool provides

  • General features

            The first tab page on the WSE Setting tool contains two check boxes. 

                                    Figure 3: General tab

    1. Enable this project for Web Services Enhancements: Checking this box will add a reference to Microsoft.Web.Services2.dll library to this Visual Studio project. Web References that are added to or updated in this project will be WSE enabled. Unchecking this item will remove this reference as well as WSE related settings from the configuration file; Web References will need to be manually updated to remove the WSE dependencies.  
    2. Enable Microsoft Web Services Enhancement Soap Extensions : Checking this box will add the WSE SOAP Extension to this project. This is only applicable to ASP.NET projects.   The following element will be added to the web.cofig file if this box is checked.

<system.web>
  <webServices>
    <soapExtensionTypes>
      <add type="Microsoft.Web.Services2.WebServicesExtension, Microsoft.Web.Services2, Version=2.0.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35" priority="1" group="0" />
    </soapExtensionTypes>
  </webServices>
</system.web>

  • Security

                                   Figure 4: Security tab

    1. Binary Security Tokens: The binarySecurityTokenManager element defines a user supplied class that supports the Microsoft.Web.Services2.Security.Tokens.ISecurityTokenManager interface to manage the authentication and loading of binary security tokens from soap envelopes and from policy documents. Multiple security managers may be declared but each must be registered for a unique value type.

      <microsoft.web.services2>
        <security>
          <binarySecurityTokenManager type="GxaToken.GxaBinaryToken, gxatoken" xmlns:gbt="http://tempuri.org/gxabinarytoken/" valueType="gbt:GxaBinaryToken"  />
        </security>
      </microsoft.web.services2>
    2. Security Tokens: The securityTokenManager element defines a user supplied class that supports the Microsoft.Web.Services2.Security.Tokens.ISecurityTokenManager interface to manage the authentication and loading of XML security tokens from soap envelopes and from policy documents. Multiple security managers may be declared but each must be registered for a qualified name. This qualified name is interpreted as the name of the xml element which defines the security token in the header.

      <microsoft.web.services2>
        <security>
          <securityToken type="GxaToken.GxaToken, gxatoken" xmlns:gbt="http://tempuri.org/gxatoken/" valueType="gbt:GxaToken"  />
        </security>
      </microsoft.web.services2>
    3. X. 509 Certificate Settings: There are four check boxes and one combo box in this section to allow developers select different attribute values for x509 element.  The x509 element defines configuration settings for the WSE X.509 certificate support. The storeLocation attribute defines which store the WSE should search when attempting to verify a certificate for a trusted root or when attempting to retrieve a certificate by name. Values for this attribute can be either "LocalMachine" or "CurrentUser". Typically, client-side uses "CurrentUser" whilst server-side uses "LocalMachine". The default is "LocalMachine". The verifyTrust property determines whether the WSE verifies that certificates used to sign a message have an issuer chain that extends to a trusted root authority. Values for the attribute are either "true" or "false" with the default as "true". The allowTestRoot property modifies the trust verification process to allow certificates signed by a test root to pass the verification. Values for this attribute are either "true" or "false". The allowRevocationUrlRetrieval corresponds to CERT_CHAIN_REVOCATION_CHECK_CACHE_ONLY flag.  This attribute value decided whether we will do URL retrieval in the revocation checking.  When it is false, revocation checking only accesses cached URLs and the WSE does not hit the wire to do any revocation URL retrieval. The allowUrlRetrieval corresponds to CERT_CHAIN_CACHE_ONLY_URL_RETRIEVAL flag. When this flag is false, only cached URLs are used in building a certificate trust chain and the WSE does not hit the wire to do any URL retrieval.  The true or false value can be selected by checking or unchecking those check boxes in this section.  The following element is example which might be added to the configuration file as an X509 element, and the attribute values shown below are all default values.  

      <microsoft.web.services2>
        <security>
          <x509 storeLocation="LocalMachine" verifyTrust="true" allowTestRoot="false" allowRevocationUrlRetrieval="true" allowUrlRetrieval="false" />
        </security>
      </microsoft.web.services2>

  • Policy

WS-Policy specific configurations. This section is not required for a default installation.  It may be used to enable WS-policy for the application. 

                                   Figure 5: Policy tab

 

    1. Enable policy : This creates a policy file for the application. 

      <microsoft.web.services2>
        <policy>    
            <cache name="policyCache.xml" />
        </ policy>
      </microsoft.web.services2>

    2. Edit Application policy : This is used to manage policies in your application.  
  • TokenIssuing

Token Issuing service related specification. This section is not required for a default installation. However, user is strongly recommended to provide a &ls;serverToken%gt; section to sign the response if this server provides token issuing service. This section is not required for a default installation. It may be used to customize token issuing service.

                                   Figure 6: TokenIssuing tab

    1. Server Token : The severToken section defines a user supplied security Token Information for a server to specify a token to sign a RequestSecurityTokenResponse. A SecurityTokenReference could be one example of keyInfoClause.  

      <microsoft.web.services2>
        <tokenIssuer>
          <serverToken>
                 <KeyInfo>
               <SecurityTokenReference>
                    <KeyIdentifier>Your Certificate's Identifier encoded in base64</KeyIdentifier>
               </SecurityTokenReference> 
            </KeyInfo> 
          </serverToken> 
        </ tokenIssuer>
      </microsoft.web.services2>

    2. ttlInSeconds: The ttlInSeconds section defines a user supplied default time-to-live for an IIssuedToken. It is optional, and its default value is 4 hours


      <microsoft.web.services2>
        <tokenIssuer>
                    <ttlInSeconds>120</ttlInSeconds>
        </ tokenIssuer>
      </microsoft.web.services2>

  • Custom filters

Filter chain configuration. This section is not required for a default installation. It may be used to install customized WSE filters.

                                   Figure 7: Custom filters tab

    1. Input filters: Click on the Add/Edit/Remove button to add/edit/remove a user supplied class type.  The input should look like "MyFilterNamespace.MyFilterClass, MyFilterDll".  The following element is an example which might be added to the configuration file to register an input filter. 

      <microsoft.web.services2>
        <filters>
          <add type="MyFilterNamespace.MyFilterClass, MyFilterDll"  />
        </ filters>
      </microsoft.web.services2>

    2. Output filters: Click on the Add/Edit/Remove button to add/edit/remove a user supplied class type.   The input should look like "MyFilterNamespace.MyFilterClass, MyFilterDll".  The following element is an example which might be added to the configuration file to register an output filter.

      <microsoft.web.services2>
        <filters>
          <add type="MyFilterNamespace.MyFilterClass, MyFilterDll" />
        </filters>
      </microsoft.web.services2>

The custom filters are run before the built-in WSE filters get run on the sending side, while they are run after the built-in WSE filters on the receiving side.  The WSE built-in filters are closest to the wire on both sending and receiving sides. 

  • Routing

                                     Figure 8: routing tab

  1. Routing Handlers:This section allows developers to define the WSE HttpHandler which implements a WS-Routing compliant SOAP router.  The verb attribute specifies that the router handles ALL incoming requests for resources that match its value. The path attribute should be modified to use a more specific path value for the URL that the router will service. Multiple entries are possible allowing the router to replace an existing Web service and delegate that service to a different URL.  The following element is an example which may be added in the configuration file to register a router.

    <system.web>
      <httpHandlers>
        <add verb="*" path="*.ashx" type="Microsoft.Web.Services2.Routing.RoutingHandler, Microsoft.Web.Services2, Version=1.0.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35" />
      </httpHandlers>
    </system.web>

  2. Referral Cache: The referralCache element points to the referral cache file that is used to make routing decisions. If this section is not provided, the default file name of "referralCache.config" is used and the file will reside in the applications base directory. If this section is provided, the file specified must exists before invoking WSE runtime. Exception will be raised otherwise.  The following element is an example which may be added in the configuration file to register a referral cache file.

    <microsoft.web.services2>
      <referral>
        <cache name="referralCache.config" />
      </referral>
    </microsoft.web.services2>

  • Diagnostics

                                 Figure 9: diagnostics tab

    1. Tracing: enabled is a required attribute for trace element, while input and output are not required. Note that enabling Trace has a significant performance penalty and should only be used for development purposes. If input and output attributes are not provided, the default file name of "inputTrace.webinfo" and "outputTrace.webinfo" are used and the file will reside in the applications base directory.  The following element is an example which may be added in the configuration file to enable trace. 

      <microsoft.web.services2>
        <diagnostics>
          <trace enabled="false" input= "inputTrace.webinfo" output= "outputTrace.webinfo" />
        </diagnostics>
      </microsoft.web.services2>

    2. Error reporting mode:There are three error reporting mode values defined in ASP.net which control the display of user-friendly error messages to users instead of error details (including a stack trace).  WSE will reuse this setting to decide whether server will send back the whole stack trace or not when exception happens. "On" Always display custom (friendly) messages; "Off" Always display detailed ASP.NET error information; "RemoteOnly" Display custom (friendly) messages only to users not running on the local Web server.  "RemoteOnly" is recommended for security purposes, so that you do not display application detail information to remote clients.  The following element is an example which may be added in the configuration file to set error reporting mode. 

<system.web>
    <customErrors mode="RemoteOnly"  />
</system.web>
